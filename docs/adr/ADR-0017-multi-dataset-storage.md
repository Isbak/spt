# ADR-0017: Multi-Dataset Storage Topology (system / agents+lineage / business)

## Status
Accepted

## Context
The platform served every named graph from a single Fuseki dataset and ingested every
R2RML mapping from a single relational source (`SOURCE_DATABASE_URL`). "Local vs remote"
was therefore all-or-nothing on both sides: a single `FUSEKI_*` pointed every graph at one
Jena, and a single `SOURCE_*` pointed every mapping at one warehouse. There was no way to
place the platform's own model, the agent registry/memory/lineage, and the business/domain
data — or their warehouses — on different stores. ADR-0007 already separates these concerns
*logically* (named graphs); ADR-0012 made source materialization drop-in but single-source;
ADR-0016 relocated authored domain content to per-domain git repos. The missing piece was a
*physical*, independently placeable split of the served graphs and their sources.

## Decision
Introduce **three storage roles**, each an independently placeable (local or remote) store
on both the read and write side:

- **system** — the platform's own model: `ontology`, `governance`, `reasoning`, `inferred`,
  `validation`. Authored/generated by the platform itself; **no relational warehouse**.
  Stays local.
- **agents** — agent registry/memory/observations (`agents`) + PROV-O lineage
  (`provenance`). Local or remote.
- **business** — domain/reference/instance data: `reference`, `masterdata`,
  `transactional`, `integration`, `sandbox`. Local or remote.

Implementation:

- **Config bundles.** `config.Settings` exposes a `FusekiDataset` bundle per role
  (`fuseki_system/agents/business`) and a `SourceDatabase` bundle per ingesting role
  (`source_business/agents`). Per-field env precedence is `FUSEKI_<ROLE>_*` /
  `SOURCE_<ROLE>_*` → shared `FUSEKI_*` / `SOURCE_*` → default, so an existing single
  `FUSEKI_BASE_URL` + `SOURCE_DATABASE_URL` keeps every role co-located. The flat
  `fuseki_*` / `source_*` names remain as aliases to the system/business bundles
  (backward compatible).
- **One routing rule, applied symmetrically.** `named_graphs.dataset_for_graph(graph)`
  maps a named graph to its role (unknown → `business`). A mapping's `map:targetGraph`
  therefore selects **both** the warehouse it reads from **and** the Fuseki dataset it is
  served to. Any warehouse-materializable graph must belong to a role that has a source,
  which is why `reference` is a business graph rather than a system graph.
- **Dataset-aware client.** `FusekiClient(dataset=<role>)` / `FusekiClient.for_graph(uri)`
  bind to a role's endpoint; `materialize`, `api.upload_default_graphs`,
  `api.fuseki_graph_triple_counts`, and `scripts/load-fuseki.sh` route per graph and treat
  each dataset's availability independently (best-effort, skip-when-down).
- **Governed manifest.** Every graph in `rdf/graphs/manifest.ttl` declares
  `gov:storedInDataset "system|agents|business"`, enforced by SHACL
  (`named_graph_shapes.ttl`) and the named-graph metadata validation gate, keeping the
  manifest and the routing map in sync.
- **Provisioning.** The bundled Fuseki container creates all three datasets
  (`FUSEKI_DATASET_1/2/3`); placing a role remotely is configuration only
  (`FUSEKI_<ROLE>_BASE_URL`, `SOURCE_<ROLE>_DATABASE_URL`).

## Consequences
- Business and agent data (and their warehouses) can live on separate, remote stores while
  the platform's own model stays local — multi-tenant scale, separate retention/backup, and
  independent access control per role.
- Existing single-store deployments are unaffected: one `FUSEKI_BASE_URL` /
  `SOURCE_DATABASE_URL` collapses all roles back to one dataset/warehouse.
- Cross-role SPARQL across datasets becomes federated (`SERVICE`) rather than cross-graph.
  This is acceptable because validation, reasoning, and tests remain **local-first** over
  the `rdf/` files (rdflib/pyshacl in-process) and do not query across datasets.
- The role of a graph is now a governed fact (`gov:storedInDataset`), so topology is
  queryable and validated rather than implicit.
